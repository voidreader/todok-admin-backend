Recommended Model : Claude Sonnet

** 한국어 스타일 유지 **

## 언제 사용하나요?

- 자동으로 사용되지 않도록 한다.
- 반드시 사용자의 호출에 의해서만 실행된다.
- `implement-agent` 스킬 이후에 사용된다.
- 명세서 없는 소규모 작업(버그 수정, 설정 조정 등)은 `finalize-minor-task`를 사용한다.

## Instructions:

### 1. 현재 작업 변경 사항 분석

- `git status` 로 수정/추가된 파일 목록 확인
- `git diff` 로 변경 내용 파악
- 구현된 기능 단위를 추론한다

### 2. 마무리 전 점검 (백엔드)

커밋 전에 아래를 점검하고, 미흡하면 사용자에게 알린다.

- **타입 체크**: `npx tsc --noEmit` 가 통과하는지 확인한다.
- **테스트**: 단위 테스트(`npm test`, Vitest면 `npx vitest run`)가 통과하는지 확인한다.
  e2e 테스트(`*.e2e-spec.ts`)는 별도로 통과 여부를 확인한다.
- **마이그레이션**: `Docs/spec/migrations/` 에 이번 작업의 SQL 파일이 있으면, **적용
  여부를 사용자에게 확인 안내**한다. 이 스킬은 마이그레이션을 직접 실행하지 않는다.
- **환경변수**: 새 환경변수를 추가했다면 `.env.example`(또는 동등 파일)이 갱신됐는지 확인한다.

점검 결과 실패 항목이 있으면 커밋을 진행하기 전에 사용자에게 보고하고 진행 여부를 확인한다.

### 3. 커밋 scope 결정

다음 우선순위로 scope를 결정한다:

1. `Docs/spec/` 에서 `{specBase}_plan.md` 파일을 찾아 명세서 이름에서 scope 추출
2. plan 파일이 없으면 `Docs/spec/` 의 명세서 파일명에서 추출 (예: `[spec]20260602_refund-system.md`)

### 4. 커밋 메시지 생성

기존 커밋 히스토리의 스타일을 확인하고, 변경 성격에 맞는 prefix를 선택한다:

- `feat:` — 새 기능 추가
- `fix:` — 버그 수정
- `refactor:` — 리팩토링
- `chore:` — 빌드, 설정, 기타

형식:

```
{prefix}: {scope} 구현 — {핵심요약1/핵심요약2/핵심요약3}

- 구현 상세 내용
- 구현 상세 내용
- 구현 상세 내용
```

작성 규칙:

- 요약 라인은 3~7개의 핵심 항목을 "/" 로 구분한다.
- Bullet 목록은 실제 구현 내용만 작성한다.
- 변경되지 않은 내용은 절대 작성하지 않는다.
- 기술적인 용어를 유지한다.
- 한국어 스타일을 유지한다.
- **AI 서명·자동 생성 트레일러(Co-Authored-By 등)를 넣지 않는다.** 사람이 직접 쓴 것처럼 자연스럽게 작성한다.

### 5. CLAUDE.md 업데이트 (조건부)

다음 **갱신 트리거 체크리스트** 중 하나라도 해당하면 CLAUDE.md를 업데이트한다:

- [ ] 새 모듈 추가 또는 모듈 경계/계층 구조 변경
- [ ] 새 엔티티/테이블 추가 또는 스키마 변경 (마이그레이션 동반)
- [ ] 새 외부 연동(외부 API·큐·캐시) 추가
- [ ] 인증/권한 정책 변경
- [ ] 새 환경변수/설정 키 추가
- [ ] 공개 API 계약 변경 (엔드포인트 추가·변경·삭제)
- [ ] 코딩 컨벤션 또는 금지 사항 변경

해당하지 않으면 CLAUDE.md를 수정하지 않는다 (단순 버그 수정, 소규모 기능 추가, 내부 리팩토링 등).

업데이트 규칙:

- 파일 전체를 수정하지 말고, 관련 섹션을 찾아 업데이트하거나 append 한다.

### 6. CHANGELOG Fragment 생성

**Docs/CHANGELOG.md를 직접 수정하지 않는다.** 대신 개별 fragment 파일을 생성한다.

파일 위치: `Docs/changelog-fragments/`

파일명: `YYYYMMDD_{scope}.md`

- 예: `20260602_refund-system.md`

내용 포맷:

```markdown
### {제목}

- 변경 내용 1
- 변경 내용 2
```

작성 규칙:

- API 소비자/운영 관점에서 간단히 요약
- 기술 용어 유지
- 한국어 스타일 유지

> fragment 합산은 `merge-changelog` 스킬로 별도 수행한다.

### 7. 문서 아카이브

최종 아카이브 구조:
```
Docs/Archive/{날짜_주제}/
├── design.md       ← 기획서 (plan-writer 산출물, 있는 경우)
├── spec.md         ← 명세서 (spec-writer 산출물)
├── plan.md         ← 구현 계획 (implement-agent 산출물)
└── migration.sql   ← 마이그레이션 SQL (있는 경우)
```

**7-1. 대상 파일 수집**

`Docs/spec/` 에서 명세서와 plan 파일을 찾는다:
- `[spec]{날짜}_{주제}.md` — 명세서
- `[spec]{날짜}_{주제}_plan.md` — 구현 계획 문서

명세서 파일을 Read로 읽어 상단 메타데이터를 확인한다:
- `> 요구 문서: {경로}` 라인을 파싱하여 기획/요구 문서 경로를 추출한다.
- `> 마이그레이션 SQL: {경로}` 라인을 파싱하여 SQL 파일 경로를 추출한다 ("없음"이면 생략).
- 각 파일이 실제로 존재하는지 확인한다. 없으면 사용자에게 알리고 해당 파일 없이 진행한다.

**7-2. 아카이브 폴더명 결정**

`{specBase}`에서 `[spec]` prefix를 제거한 값을 폴더명으로 사용한다.
- 예: `[spec]20260602_refund-system` → 폴더명 `20260602_refund-system`

**7-3. 아카이브 수행**

1. `Docs/Archive/{폴더명}/` 폴더를 생성한다.
2. 기획/요구 문서가 있으면 → `design.md`로 이름 변경하여 아카이브 폴더에 저장한다.
3. `{specBase}.md` → `spec.md`로 이름 변경하여 아카이브 폴더에 저장한다.
4. `{specBase}_plan.md` → `plan.md`로 이름 변경하여 아카이브 폴더에 저장한다.
5. 마이그레이션 SQL이 있으면 → `migration.sql`로 이름 변경하여 아카이브 폴더에 저장한다.
6. 원본 파일들을 모두 삭제한다 (`git rm` 사용):
   - `Docs/spec/{specBase}.md`
   - `Docs/spec/{specBase}_plan.md`
   - `Docs/spec/migrations/[migration]{날짜}_{주제}.sql` (있는 경우)
   - 기획/요구 문서 원본 파일 (있는 경우)

**아카이브 문서 갱신**: 구현 이후 사용자 테스트 피드백, 오류 수정, 추가 구현 등이 진행된 경우, 아카이브 문서를 업데이트한다:

- **plan.md**: `## 추가 변경 사항` 섹션을 하단에 추가한다. 기존 내용은 유지하고, 이후 변경된 파일 목록과 수정 내역을 추가 기재한다.
- **spec.md**, **design.md**, **migration.sql**: 원본이므로 수정하지 않는다.

### 8. git 커밋 수행

- 변경된 파일을 **개별적으로 스테이징**한다 (`git add .` 사용 금지).
  - 예: `git add src/ Docs/ CLAUDE.md`
  - `.gitignore`에 포함되지 않는 임시 파일이나 민감 파일이 섞이지 않도록 주의한다.
  - `.env` 파일은 절대 스테이징하지 않는다.
- **HEREDOC을 사용하여** 멀티라인 커밋 메시지를 작성한다:

```bash
git commit -m "$(cat <<'EOF'
feat: {scope} 구현 — {핵심요약1/핵심요약2}

- 상세 내용
EOF
)"
```

중요:

- **git push 는 절대 실행하지 않는다.**
- commit 까지만 수행하고 종료한다.

### 9. 결과 출력

사용자에게 다음을 출력한다:

- 생성된 커밋 메시지
- 커밋에 포함된 파일 목록
- push가 수행되지 않았음을 명시
- **마이그레이션 SQL이 아직 적용되지 않았다면 그 사실과 적용 안내**
- 새 환경변수 추가 시 배포 환경 설정 갱신 안내
