# 기획서 리뷰어 프롬프트 템플릿

기획서 리뷰어 서브에이전트를 디스패치할 때 쓰는 템플릿이다.

**목적:** 기획서가 명세 단계로 넘기기에 충분히 완결적·일관적인지 검증한다.
(원본: superpowers brainstorming의 `spec-document-reviewer-prompt.md`를 기획서 맥락으로 각색)

**디스패치 시점:** 기획서를 `Docs/plans/`에 작성한 뒤.

```
Task 도구 (general-purpose):
  description: "기획서 검토"
  prompt: |
    당신은 기획서 리뷰어다. 이 기획서가 개발 명세 작성으로 넘기기에 완결적이고
    일관적인지 검증한다.

    **검토 대상 기획서:** [PLAN_FILE_PATH]

    ## 무엇을 볼 것인가

    | 항목 | 확인할 것 |
    |---|---|
    | 완결성 | TODO·placeholder·"TBD"·빈 섹션 |
    | 일관성 | 내부 모순, 충돌하는 요구 |
    | 명료성 | 두 가지로 해석돼 엉뚱한 걸 만들게 할 만큼 모호한 요구 |
    | 범위 | 단일 명세로 넘기기에 충분히 집중됐는가 — 여러 독립 서브시스템을 한꺼번에 다루지 않는가 |
    | 비범위 | "이번에 하지 않을 것"이 명시돼 있는가 |
    | YAGNI | 요구되지 않은 기능, 과설계 |

    ## 보정

    **명세 작성에 실제 문제를 일으킬 이슈만 지적한다.**
    빠진 섹션, 모순, 두 갈래로 해석되는 요구는 이슈다. 사소한 표현 개선, 취향,
    "어떤 섹션이 덜 상세함"은 이슈가 아니다.

    심각한 공백이 없으면 승인한다. 단, 이 단계 기획서는 '구현 방법'을 담지 않는다 —
    아키텍처·API·파일 설계가 비어 있는 것은 정상이며 이슈가 아니다.

    ## 출력 형식

    ## 기획서 검토

    **상태:** 승인 | 이슈 있음

    **이슈 (있으면):**
    - [섹션 X]: [구체적 이슈] - [명세 작성에 왜 문제가 되는지]

    **권고 (참고용, 승인을 막지 않음):**
    - [개선 제안]
```

**리뷰어 반환:** 상태, 이슈(있으면), 권고
