# `google-sheets-safe-edit` 스킬

Google Sheets에 대한 모든 수정 요청을 받았을 때, 실제 쓰기 전에 영향 범위를 투명하게 보여주고 명시적 승인을 받는 절차를 강제한다. 잘못된 시트/범위/값으로의 덮어쓰기를 차단하고, 실수 발생 시 즉시 복구 가능한 백업 스냅샷을 남기는 것이 목적이다.

## 왜 필요한가

구글 시트 수정은 되돌리기가 불편하다. 특히 다음 상황에서 피해가 크다:
- 시트 이름 오타로 엉뚱한 시트를 덮어씀
- 범위 계산 실수로 의도보다 넓은 영역을 덮어씀
- `share_spreadsheet`로 권한을 실수로 확장
- 셀 타입을 문자열에서 숫자로 바꿔 수식이 깨짐

이 스킬은 "쓰기 전 한 번의 정지"를 제도적으로 보장한다. LLM의 자체 판단에 맡기지 않는다.

## 적용 대상 (write 도구)

아래 MCP 도구를 호출하기 전에는 반드시 이 스킬의 전체 흐름(1~5단계)을 따른다:

- 셀 편집: `mcp__google-sheets__update_cells`, `batch_update_cells`, `batch_update`
- 구조 추가: `mcp__google-sheets__add_rows`, `add_columns`
- 시트 조작: `mcp__google-sheets__create_sheet`, `rename_sheet`, `copy_sheet`
- 스프레드시트: `mcp__google-sheets__create_spreadsheet`
- 시각화: `mcp__google-sheets__add_chart`
- 권한: `mcp__google-sheets__share_spreadsheet` — 민감 작업

## 제외 대상 (read-only, 승인 불필요)

아래는 이 스킬을 거치지 않는다:

- `list_spreadsheets`, `list_sheets`, `list_folders`
- `get_sheet_data`, `get_multiple_sheet_data`, `get_multiple_spreadsheet_summary`, `get_sheet_formulas`
- `find_in_spreadsheet`, `search_spreadsheets`

## 작업 흐름

### 1단계 — 수정 계획 수립

요청을 분해하여 다음 네 가지를 명확히 한다:

- 대상 spreadsheet ID 또는 이름
- 대상 시트 이름
- 수정 범위 (셀/행/열/시트 전체/스프레드시트 단위)
- 변경 의도 (예: "PlayerStat 시트의 Level 3 행 HP 값을 1000→1500")

정보가 부족하면 먼저 read-only 도구(`list_sheets`, `get_sheet_data` 등)로 조사한다. 추측으로 진행하지 않는다.

### 2단계 — 현재 상태 조회 및 백업 저장

실행 전 영향 범위의 현재 값을 반드시 읽는다:

- 셀/범위 수정: `get_sheet_data`로 해당 범위 조회
- 시트 이름 변경/삭제/복사: 필요한 범위를 `get_sheet_data`로 덤프
- 권한 변경: 현재 공유 상태를 파악 가능한 선까지 확인

백업 파일을 아래 규칙으로 저장한다:

- **경로**: 프로젝트 루트 기준 `.Codex/google-sheets-backups/`
  - 프로젝트 루트는 Codex 세션의 현재 작업 디렉토리(`$CWD` / `pwd`)로 판단한다
  - 디렉토리가 없으면 생성한다
  - `.gitignore`에 `.Codex/google-sheets-backups/` 항목이 없다면 자동 추가를 제안한다 (백업은 로컬 복구용이며 커밋 대상이 아니다)
- **파일명**: `YYYYMMDD-HHMMSS_<spreadsheetId앞8자>_<sheet>_<range요약>.json`
  - 파일명에 사용할 수 없는 문자는 `_`로 치환
- **내용** (JSON):
  ```json
  {
    "timestamp": "2026-04-17T15:03:21+09:00",
    "tool": "mcp__google-sheets__update_cells",
    "spreadsheetId": "...",
    "spreadsheetName": "...",
    "sheet": "...",
    "range": "A1:C5",
    "before": [ ... 현재 값 ... ],
    "planned": [ ... 변경 후 값 ... ],
    "reason": "<사용자 요청 요약>"
  }
  ```

**로테이션**: 백업 디렉토리의 `.json` 파일이 20개를 초과하면, 파일명 오름차순(= 시간순)으로 정렬해 가장 오래된 것부터 20개 이하가 될 때까지 삭제한다.

백업 저장에 실패하면 3~5단계를 진행하지 않고 사용자에게 오류를 보고한다.

### 3단계 — 변경 프리뷰 작성

아래 템플릿에 맞춰 출력한다. 섹션을 생략하거나 순서를 바꾸지 않는다. 백업 경로는 **프로젝트 루트 기준 상대경로**로 표시한다 (절대경로 금지 — 팀원마다 OS/경로가 다르다).

```
## Google Sheets 수정 승인 요청

**대상**: <스프레드시트 이름 또는 ID 앞 8자> / `<시트 이름>` / 범위 `<A1:C5>`
**작업**: <tool 이름>
**영향 셀 수**: <N>
**백업 경로**: `.Codex/google-sheets-backups/<filename>.json`

### 변경 전 (before)
| cell | value |
|------|-------|
| A1   | 100   |

### 변경 후 (after)
| cell | value |
|------|-------|
| A1   | 200   |

위와 같이 수정하려 합니다. 진행할까요? (네/OK/진행)
```

규모별 출력 원칙:

- **20셀 이하**: before/after 전부 표시
- **21~200셀**: 앞 10 + 뒤 5만 표시하고 중간은 `... (M개 생략) ...`로 줄임
- **200셀 초과**: 요약(총 변경 셀 수, 무작위 샘플 5개)만 표시하고 "전체 목록이 필요하면 요청하세요" 안내
- **시트 구조 변경** (create/rename/copy): 표 대신 텍스트 요약 (예: "`Stage` 시트를 `Stage_Old`로 이름 변경", "`PlayerStat` 시트를 `PlayerStat_copy`로 복제")
- **스프레드시트 생성**: 생성될 파일 이름, 대상 폴더, 초기 시트 목록을 명시
- **공유 권한 변경**: **반드시 전체 내역을 표시** — 추가/변경될 이메일, 권한 레벨(reader/writer/commenter), 링크 공유 여부 명시. 요약 금지

민감 작업은 추가 라벨을 붙인다:

- `share_spreadsheet` 호출: 프리뷰 상단에 `민감 작업: 권한 변경` 라벨
- 100셀 초과 수정: 상단에 `대규모 수정 (N셀)` 라벨
- 수식 포함 셀 덮어쓰기: 상단에 `수식 덮어쓰기 포함` 라벨

### 4단계 — 승인 대기

사용자의 다음 입력을 받을 때까지 write 도구를 호출하지 않는다.

**승인으로 판정할 자연어 표현**:

- "네", "예", "응", "어", "좋아", "좋습니다", "OK", "ok", "진행", "진행해", "해줘", "go", "yes", "y", "approve", "승인"
- 위에 직접 없더라도 명확히 긍정 의도가 드러나는 자연어는 승인으로 본다 (예: "그렇게 해줘")

**승인이 아닌 응답**:

- 질문, 추가 지시, "잠깐", "잠시만", "다시", "아니", "no", "취소", "stop", "멈춰"
- 무응답 상태로 판단 불가하면 **절대 진행하지 않는다**

프리뷰와 다른 새 요청이 들어오면 지금까지의 프리뷰는 폐기하고 1단계로 돌아간다.

### 5단계 — 실행 및 결과 확인

승인 후에만 해당 MCP write 도구를 호출한다. 실행 이후:

- **성공 시**: 변경된 범위를 `get_sheet_data`로 재조회해 after 값이 반영됐는지 확인한 뒤 아래 한 줄 형식으로 보고 (경로는 프로젝트 루트 기준 상대경로)
  ```
  완료: `<sheet>!<range>` (<N>셀) · 백업: `.Codex/google-sheets-backups/<filename>.json`
  ```
- **실패 시**: 오류 메시지와 함께 백업 경로를 안내하고 롤백 방법을 제시
- **부분 실패** (batch 도구에서 일부 항목만 실패): 성공/실패 항목을 분리해서 보고

## 다중 수정 요청 처리

한 사용자 요청으로 여러 쓰기 작업이 필요한 경우:

- 서로 연관되고 같은 맥락(같은 시트, 같은 목적)인 작업들은 **하나의 프리뷰로 묶어** 일괄 승인 받는다
- 서로 무관한 작업(다른 시트, 다른 목적)은 작업별로 각각 프리뷰 → 승인을 반복한다
- `batch_update` / `batch_update_cells` 내부의 개별 항목도 모두 diff를 표시한다. 배치라고 요약하지 않는다

## 롤백

수정 완료 보고에는 항상 백업 파일 경로(상대경로)를 포함한다. 사용자가 "되돌려"/"롤백" 요청 시:

1. 백업 JSON의 `before` 값을 읽는다
2. 이를 기준으로 역수정 계획을 세운다
3. **본 스킬의 1~5단계를 다시 그대로 거친다** (롤백도 쓰기 작업이므로 승인 필요)

## 실수 방지 원칙

- 요청을 받자마자 write 도구를 호출하지 않는다. 2~3단계를 반드시 선행한다
- 프리뷰의 after 값이 사용자 의도와 맞는지 스스로 한 번 더 검산한다 (타입, 단위, 시트 이름 오타, 범위 off-by-one)
- 백업 저장 실패 시 쓰기 중단
- 경로는 항상 프로젝트 루트 기준 상대경로로 표시. 절대경로 금지 (팀원 간 공유되는 출력이므로)
- "일부만 빠르게 수정"하라는 요청도 예외 없이 전체 흐름을 따른다 — 규모 판단은 프리뷰 출력 방식(상세/요약)에서만 반영

## 자주 하는 실수

- **요청에 spreadsheet ID가 있다고 바로 write**: ID가 있어도 시트 이름·범위가 모호하면 먼저 조사한다
- **프리뷰 없이 "수정하겠습니다"만 말하고 호출**: 프리뷰는 구조화된 표로 제시해야 한다. 자연어 요약은 프리뷰가 아니다
- **승인으로 받은 "확인"을 행동 개시 신호로 오해**: "확인"은 모호하다. "네/OK/진행" 계열이 아니면 재확인한다
- **batch 도구에서 diff 생략**: 내부 모든 항목을 표시한다. 항목이 많으면 규모별 출력 원칙(21~200셀, 200셀 초과)을 적용한다
- **백업 경로를 절대경로로 표기**: 타 팀원 환경과 불일치. 반드시 프로젝트 루트 기준 상대경로
