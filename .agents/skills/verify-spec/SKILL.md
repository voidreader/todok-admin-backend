description: 명세서가 기획 문서의 의도를 정확히 반영했는지 검증한다. 5개 항목을 체크하여 PASS 또는 FAIL + 이슈 목록을 반환한다. spec-pipeline에서 자동 호출되며, 단독 호출도 가능하다.

Recommended Model : Codex Opus
** 한국어 스타일 유지 **

## 언제 사용하나요?

- `spec-pipeline` 오케스트레이터에서 Opus sub-agent로 자동 호출된다.
- 명세서를 단독으로 재검증할 때 직접 호출할 수 있다.
- 호출 예: `/verify-spec @Docs/content-design/[content]20260414_plan.md @Docs/spec/[spec]20260414_spec.md`

## Instructions

### 입력

- 첫 번째 인자: 기획 문서 경로
- 두 번째 인자: 명세서 경로

두 인자 중 하나라도 없으면 "기획 문서와 명세서 경로를 모두 전달해주세요." 메시지를 출력하고 중단한다.

### 수행 절차

**1단계: 문서 읽기**

기획 문서와 명세서를 Read로 읽는다.

**2단계: 5개 항목 검증**

아래 항목을 순서대로 검증한다.

**[항목 1] 기획 의도 반영도**
- 기획서의 모든 주요 요구사항이 FR 항목으로 변환됐는가?
- 기획서에서 명시한 핵심 동작이 누락 없이 명세서에 반영됐는가?
- 기획서의 의도와 다르게 해석된 FR 항목이 없는가?

**[항목 2] FR 명확성**
- 모호한 표현("적당히", "빠르게", "필요 시" 등)이 구체적 조건/수치로 대체됐는가?
- 각 FR 항목에 상세 동작과 조건이 명시됐는가?
- 복수로 해석 가능한 문장이 없는가?

**[항목 3] 수치·공식 정확성**
- 밸런스 수치, 계산 공식, 상수값이 기획서와 일치하는가?
- 단위(분, 골드, %, Lv 등)가 명확히 표기됐는가?
- 기획서에 있는 수치가 명세서에서 임의로 변경되지 않았는가?

**[항목 4] 시스템 상호작용 완결성**
- 관련 시스템(Provider, Service, Repository)과의 상호작용이 모두 명시됐는가?
- 영향 범위(수정/신규/삭제 파일)가 빠짐없이 기재됐는가?
- 엣지 케이스(예외 상황, 동시 발생 조건)가 처리됐는가?

**[항목 5] verifier 검증 가능성**
- 각 FR 항목이 코드를 보고 PASS/FAIL을 판정할 수 있는 형태인가?
- "구현되어야 한다" 수준이 아닌, 구체적인 동작 기준이 있는가?
- verifier가 체크할 수 없는 모호한 FR이 없는가?

**3단계: 결과 출력**

아래 형식으로 출력한다.

5개 항목이 모두 PASS인 경우에만 최종 PASS 판정한다.
1개라도 FAIL이면 최종 FAIL 판정한다.

```
## verify-spec 결과: [PASS / FAIL]

### 검증 항목

| # | 항목 | 결과 | 비고 |
|---|------|------|------|
| 1 | 기획 의도 반영도 | PASS/FAIL | |
| 2 | FR 명확성 | PASS/FAIL | |
| 3 | 수치·공식 정확성 | PASS/FAIL | |
| 4 | 시스템 상호작용 완결성 | PASS/FAIL | |
| 5 | verifier 검증 가능성 | PASS/FAIL | |

### 이슈 목록
(FAIL 항목이 있는 경우에만 작성)
- [ISSUE-1] [항목명] 구체적 문제 및 수정 방향
- [ISSUE-2] ...

### 수정 지시사항
(FAIL 시 spec-writer가 반영해야 할 사항을 구체적이고 실행 가능한 형태로 기술)
1. ...
2. ...
```
