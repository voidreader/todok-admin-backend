name: milestone-runner
description: pipeline.config.md에 정의된 N개 단계를 체크포인트 기반으로 순차 진행하는 범용 파이프라인 러너. 작업 단위 하나를 선택하면 설정에 정의된 단계를 순서대로 안내하고, 각 단계 종료 시 사용자 승인을 받으며, 상태 파일에 진행을 기록하여 중단/재개를 지원한다. child 스킬을 직접 호출하지 않고 사용자에게 실행 명령어를 안내한다. 설정 주입형이라 도메인·스택과 무관하게 동작한다.

Recommended Model : Codex Opus
** 한국어 스타일 유지 **

## 언제 사용하나요?

- 사용자의 명시적 호출(`/milestone-runner`)에 의해서만 사용한다
- 여러 스킬을 정해진 순서로 거쳐야 하는 다단계 작업의 진입점으로 사용한다
- 단계 사이에 사람의 승인이 필요하고, 중단 후 재개가 필요한 긴 작업에 적합하다

이 스킬은 **단계 구성을 하드코딩하지 않는다.** 어떤 단계를, 어떤 스킬로, 어떤
산출물 경로로 진행할지는 모두 같은 디렉토리의 `pipeline.config.md`에서 읽는다.
프로젝트는 이 스킬을 가져온 뒤 `pipeline.config.md`를 자기 파이프라인에 맞게
채워야 한다. (채운 예시는 `examples/band-of-mercenaries/skills/milestone-runner` 참고)

## 입력 형식

```
/milestone-runner {unit} [--resume] [--phase {번호}]
```

- `{unit}` (필수): 진행할 **작업 단위** 식별자. config의 "작업 단위 소스"에서
  추출하거나, 소스가 없으면 호출 시 사용자와 대화로 정의한다
- `--resume` (선택): 기존 상태 파일을 읽어 중단된 지점에서 재개
- `--phase {번호}` (선택): 특정 단계 번호로 점프 (1~N). 체크포인트는 유지된다

호출 예:
- 신규 시작: `/milestone-runner auth-system`
- 재개: `/milestone-runner auth-system --resume`
- 특정 단계부터: `/milestone-runner auth-system --phase 3`

## 설정 파일 (pipeline.config.md)

이 스킬은 시작 시 같은 디렉토리의 `pipeline.config.md`를 Read하여 다음을 파악한다:

1. **단계 목록** — 단계마다: 이름 / 호출 스킬(명령어 형식) / 산출물 디렉토리 /
   선행 입력 참조 규칙
2. **작업 단위 소스** — 단위 목록을 읽을 문서 경로와 식별 방법 (없으면 대화 정의)
3. **상태 파일 루트 경로** — 기본 `Docs/milestone-runs/`
4. **단계 스킵 정책** (선택)

`pipeline.config.md`가 없거나 단계 목록이 비어 있으면 진행하지 않고, 사용자에게
설정 파일을 먼저 채우도록 안내한다.

## 단계 구조

단계 수와 내용은 config가 정한다. 일반 형태:

| 단계 | 이름 | 사용 스킬 | 산출물 디렉토리 |
|------|------|----------|----------------|
| 1 | (config) | (config) | (config) |
| … | … | … | … |
| N | (config) | (config) | (config) |

**핵심 원칙:**
- 각 단계 종료 시 **사용자 승인 체크포인트**에서 멈춘다. 자동으로 다음 단계로 넘어가지 않는다
- config의 스킵 정책 또는 사용자 판단에 따라 일부 단계는 스킵 가능하다
- 한 단계 내 다중 산출물을 허용하며, 의존 관계/순서를 권장한다
- child 스킬을 **직접 호출하지 않는다.** 사용자에게 실행할 명령어를 안내하고,
  사용자가 `--resume`으로 돌아오도록 유도한다

## 상태 파일

**위치**: `{상태 파일 루트}/{unit}/state.md` (루트는 config 지정, 기본 `Docs/milestone-runs/`)

**형식 (마크다운 + 체크리스트):**

```markdown
# {unit} 실행 상태

> 시작: {ISO timestamp}
> 마지막 업데이트: {ISO timestamp}
> 현재 단계: {N}
> 상태: {planned | in_progress | completed | paused}

## 작업 단위 요구사항 요약

{unit 소스에서 추출한 핵심 요구사항 3~5줄. 소스가 없으면 대화로 합의한 내용}

## 단계 1: {config의 단계 1 이름}

**상태**: pending | in_progress | completed | skipped

계획된 산출물:
- [ ] 1. {주제}
  - 참고 입력: {선행 단계 산출물 경로 또는 참고 문서}
  - 산출물: (미생성)
- [x] 2. {주제}
  - 산출물: {경로}
  - 완료: {ISO timestamp}

## 단계 2: {config의 단계 2 이름}

(동일 형식)

## 실행 이력

- {timestamp}: 작업 단위 시작
- {timestamp}: 단계 1 계획 승인
- {timestamp}: "주제A" 완료 ({경로})
- ...
```

상태 파일은 **사람도 읽을 수 있는 마크다운**으로 유지한다. 기계 파싱은
체크리스트(`- [x]` / `- [ ]`)와 섹션 헤더로 수행한다.

## 수행 절차

### 0단계: 설정 로딩 및 모드 판별

- `pipeline.config.md`를 Read하여 단계 목록·unit 소스·상태 파일 루트·스킵 정책을 파악한다
  - 설정이 없거나 단계 목록이 비어 있으면 중단하고 설정 작성을 안내한다
- `{상태 파일 루트}/{unit}/state.md` 존재 여부 확인
- `--resume` 있고 상태 파일 존재 → **재개 모드** (5단계로 이동)
- `--resume` 있는데 상태 파일 없음 → 에러 보고 후 중단
- `--resume` 없고 상태 파일 **이미 존재** → 사용자에게 확인 요청 (덮어쓸지 / 재개할지 / 중단할지)
- 그 외 → **신규 시작 모드** (1단계로 진행)

### 1단계: 컨텍스트 로딩 및 요구사항 추출

config의 "작업 단위 소스"가 지정되어 있으면 해당 문서에서 `{unit}` 섹션을 Read하여
요구사항을 추출한다. 소스가 없으면 사용자와 대화로 이 작업 단위의 목표·산출 범위를
합의한다.

추출/합의할 항목:
- 작업 단위의 목표·테마
- 포함될 작업/시스템 목록
- 종료 조건 (완료 체크리스트)
- 선행 의존성 (다른 작업 단위 결과물)

config가 추가 참고 문서를 지정했다면 함께 Read한다. **게임 전용 문서 등 config에
없는 문서를 임의로 가정하지 않는다.**

추출 결과를 사용자에게 요약 보고한다.

### 2단계: 단계별 산출물 계획 제안

config의 단계 목록을 기준으로 각 단계의 권장 산출물 목록을 제안한다.
요구사항에서 독립적 주제/시스템별로 산출물을 1개씩 분리하는 것을 권장한다.

**산출물 간 의존 관계/순서를 명시**한다. 예:
```
단계 3 산출물 1 → 입력: 단계 1 산출물 2 + 단계 2 산출물 1
```

config의 스킵 정책에 해당하거나 불필요한 단계는 사용자에게 스킵을 명시 제안한다.

제안 후 사용자 승인:
```
## 제안하는 산출물 계획

### 단계 1: {이름} (N개)
1. ...
2. ...

### 단계 2: {이름} (N개)
1. ...

### 단계 간 의존
- 단계 3 항목 1 → 단계 1 항목 2 + 단계 2 항목 1 필요

이 계획으로 진행할까요?
- (y) 승인, 상태 파일 생성 후 단계 1 시작
- (수정 내용) 추가/제거/순서 변경 요청
- (n) 중단
```

### 3단계: 상태 파일 생성

- `{상태 파일 루트}/{unit}/` 디렉토리 생성 (이미 있으면 확인)
- 승인된 계획을 바탕으로 `state.md` 작성
- 현재 단계: 1
- 모든 산출물: `pending`
- 실행 이력에 "작업 단위 시작" + "계획 승인" 기록

### 4단계: 단계 시작 안내

현재 단계(기본 1)의 **첫 번째 pending 산출물**에 대한 액션을 사용자에게 안내한다.
호출 명령어는 config의 해당 단계 "호출 스킬" 형식을 사용한다.

```
## 단계 {N}: {단계 이름} 시작

**현재 액션: {단계 내 순서}/{총개수}**

주제: {산출물 주제}
참고 입력: {선행 단계 산출물 경로 목록}

실행할 명령어:
`{config의 해당 단계 호출 스킬 형식}`

예상 산출물 경로: `{config의 산출물 디렉토리}/{날짜}_{슬러그}.md`

해당 스킬이 산출물을 생성하면 아래 명령어로 돌아오세요:
`/milestone-runner {unit} --resume`
```

**milestone-runner는 여기서 종료한다.** child 스킬은 사용자가 직접 실행한다.

### 5단계: 재개 모드 (`--resume`)

1. `state.md`를 Read
2. 현재 단계, 완료 체크 목록, 계획된 산출물 파악
3. 해당 단계의 산출물 디렉토리(config 지정)를 Glob으로 스캔
   - 상태 파일의 "마지막 업데이트" 이후 생성/수정된 파일 식별
4. 새 파일을 기존 pending 항목에 매칭 제안:
   ```
   ## 재개 상태 확인

   새로 감지된 산출물:
   - {경로}

   아래 계획 항목 중 어느 것과 매칭되나요?
   - 1. {항목명} (pending)
   - 2. {항목명} (pending)
   - (새 항목으로 추가)
   - (무시)
   ```
5. 사용자 매칭 입력 후 state.md 갱신 (체크 + 경로 + 완료 timestamp)
6. 다음 액션 결정:
   - 현재 단계에 pending 항목 남음 → 4단계 스타일로 다음 산출물 안내
   - 모든 pending 완료 → **6단계 (단계 종료 체크포인트)**

### 6단계: 단계 종료 체크포인트

```
## 단계 {N}: {단계 이름} 완료

### 생성된 산출물 ({N}개)
- {경로1}
- {경로2}

### 다음 단계 {N+1}: {다음 단계 이름} 미리보기

계획된 산출물:
- {주제1}
  - 입력: {의존 산출물 경로}
- {주제2}

### 진행 결정
- (y) 단계 {N+1} 시작
- (skip) 다음 단계 스킵 (이 작업 단위에 불필요한 경우)
- (수정) 다음 단계 산출물 계획 수정
- (중단) 여기서 중단. 상태 저장됨. 나중에 `--resume`으로 재개 가능
```

- `y`: state.md의 현재 단계를 `N+1`로 갱신 후 4단계 실행
- `skip`: 다음 단계 상태를 `skipped`로 기록 후 그 다음 단계의 종료 체크포인트로 이동
- `수정`: 계획 수정 후 다시 체크포인트 표시
- `중단`: state.md의 상태를 `paused`로 기록하고 종료. `--resume`으로 재진입 가능

마지막 단계(N) 완료 시 → **7단계**로 이동

### 7단계: 작업 단위 완료 보고

```
## {unit} 실행 완료

### 전체 산출물 요약
- 단계 1 ({이름}): {N}개
  - {경로 목록}
- 단계 2 ({이름}): {N}개
  - {경로 목록}
- ...

### 다음 단계 권장
{config가 마지막 단계 이후 권장 액션을 지정했다면 안내. 예: 생성된 명세서를
구현 스킬로 전달}

### 상태 파일 최종
`{상태 파일 루트}/{unit}/state.md`에 전체 이력이 기록되었습니다.
상태가 `completed`로 변경되었습니다.
```

## 주의사항

- **child 스킬을 직접 호출하지 않는다.** 각 child는 인터랙티브 Q&A가 많으므로
  사용자가 직접 실행해야 한다. milestone-runner는 조율자일 뿐
- **state.md가 진실의 원천이다.** 산출물 경로는 매 재개 시 파일시스템에서 재검증한다
- **설정/입력 파싱 실패 시 임의 계획을 생성하지 않는다.** 사용자에게 질문한다
- **체크포인트 없이 단계를 건너뛰지 않는다.** `--phase N`으로 명시적 점프한 경우에도
  해당 단계 시작 시점에 승인을 받는다
- **여러 사용자가 동시에 상태 파일을 수정하는 경우를 가정하지 않는다.** 단일 사용자 도구
- **자동으로 산출물 계획을 변경하지 않는다.** 재개 시 계획 항목과 매칭되지 않는 신규
  파일이 있으면 반드시 사용자에게 확인
- **config에 정의되지 않은 단계·경로·스킬을 임의로 가정하지 않는다.**
