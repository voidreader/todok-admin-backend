Recommended Model : Codex Opus
** 한국어 스타일 유지 **

## 언제 사용하나요?

- 자동으로 사용되지 않도록 한다.
- 사용자의 자의적 호출에 의해서 사용되도록 한다.
- 기획/요구 문서를 백엔드 개발 명세서로 변환할 때 사용한다.

## 이 스킬의 백엔드 특이사항 (먼저 읽을 것)

이 명세서의 하류 구현은 `implement-agent`(common 골격 + backend 어댑터)이며, backend
어댑터는 **검증모드가 `tdd`**다. 따라서 명세서는 세 가지를 반드시 만족한다:

1. **TDD 기반 명세** — 각 기능 요구사항(FR)을 검증할 테스트 케이스(정상·실패·엣지)를
   명세서의 `테스트 명세` 섹션에 미리 적는다. coder가 RED 테스트(`*.spec.ts`)를 먼저
   작성할 때 이 케이스를 그대로 입력으로 쓴다. 테스트로 표현할 수 없는 FR은 FR을
   다시 쪼개거나 "기획 확인 사항"으로 올린다.
2. **결정 지점은 선택지 + 장단점** — 사용자 결정이 필요한 모든 곳에서 옵션을 나열할
   때 옵션별 **장점·단점·권장 여부**를 표로 함께 제시한다. 결론만 통보하지 않는다.
3. **DB 마이그레이션은 SQL 파일로 분리** — 스키마 변경이 있으면 마이그레이션 SQL을
   별도 `.sql` 파일(UP/DOWN 포함)로 **작성**한다. 단, spec-writer는 그 SQL을 **직접
   실행하거나 DB에 적용하지 않는다**. 적용은 구현/배포 단계에서 사람이 한다.
4. **성능·동시성·멱등성은 요구 문서에 언급이 없어도 항상 점검** — 백엔드 기능은
   문서에 명시되지 않았더라도 부하·동시 요청·재시도 상황에서 깨질 수 있다. spec-writer는
   아래를 능동적으로 점검하되, **구현 방법을 임의로 정해 명세에 박지 않는다**. 고려가
   필요한 지점을 찾으면 사용자에게 **선택지 + 장단점**으로 제시하고 **동의를 받은 뒤**
   그 결정을 명세에 반영한다 (3단계 "의도 확인"에서 처리).
   - **성능**: N+1, 대용량 조회 페이지네이션, 인덱스 필요 여부, 무거운 동기 작업
   - **동시성**: 동시 요청 시 경합(race)·중복 처리·정합성, 트랜잭션 경계와 격리 수준,
     필요한 잠금(낙관적/비관적)
   - **멱등성**: 재시도·중복 요청에 안전한가(특히 결제·주문·외부 연동 등 부수효과 발생
     엔드포인트), 멱등성 키·중복 방지 전략 필요 여부

## Instructions

기획/요구 문서의 의도를 정확히 파악하고, 현재 코드베이스를 탐색하여 구현에 필요한
구체적인 정보와 테스트 명세를 담은 명세서를 생성한다.

# 입력

사용자가 `/spec-writer` 호출 시 인자로 기획/요구 문서 파일 경로를 전달한다.
- 호출 예: `/spec-writer @Docs/20260602_결제환불_요구사항.md`
- `plan-writer` 스킬로 작성한 기획서는 `Docs/plans/YYYY-MM-DD-<주제>.md`에 있다. 이
  파이프라인에서 넘어온 경우 인자로 그 경로가 전달된다.

전달된 인자에서 파일 경로를 추출한다. 파일 경로가 포함되지 않은 경우, 먼저
`Docs/plans/` 디렉토리를 확인한다.
- `Docs/plans/`에 기획서가 있으면 가장 최근 파일을 찾아 "이 기획서로 진행할까요?"라고
  사용자에게 확인한 뒤 사용한다.
- `Docs/plans/`가 없거나 비어 있으면 "기획/요구 문서 파일 경로를 전달해주세요. (예:
  /spec-writer @Docs/20260602_feature.md)" 메시지를 출력하고, 경로가 전달될 때까지
  진행하지 않는다.

파일 경로가 확인되면 Read로 해당 파일이 존재하는지 검증한다. 존재하지 않으면:
- "파일을 찾을 수 없습니다: {경로}" 메시지를 출력한다.
- 올바른 경로가 전달될 때까지 진행하지 않는다.

# 수행 절차

## 1단계: 요구 문서 분석

문서를 Read로 읽고 다음을 파악한다:
- 기능의 목적과 핵심 동작
- API 소비자(클라이언트·다른 서비스) 관점의 기대 동작
- 입력/출력 데이터와 검증 규칙 (있는 경우)
- 인증/권한·트랜잭션·멱등성 요구사항 (있는 경우)
- 비기능 요구사항: 성능·동시성·정합성·감사 로그 (있는 경우)
- 문서에서 모호하거나 누락된 부분

## 2단계: 코드베이스 탐색

문서의 기능과 관련된 기존 코드를 탐색한다. Agent(subagent_type="Explore")를 활용하여
병렬로 탐색할 수 있다. 다음을 **모두 확인할 때까지** 탐색을 계속한다:

1. 기능이 속하는 모듈/도메인 식별 (AGENTS.md의 아키텍처 섹션 참조)
2. 수정 대상 파일을 모두 Read로 열어 실제 구조를 확인했다 (Controller / Service /
   Repository / Module / DTO / Entity)
3. 기능의 진입점을 식별했다 — REST/GraphQL 엔드포인트(Controller·Resolver), 큐
   컨슈머, 이벤트 핸들러, 스케줄러(cron) 중 어디서 호출이 시작되는가
4. 유사 기능이 있다면 해당 구현을 Read로 확인했다
5. 새로 생성해야 할 파일을 식별했다
6. DB 변경 필요 여부를 확인했다 — 관련 엔티티/테이블, 컬럼, 인덱스, 제약, enum,
   기존 마이그레이션 파일 위치와 네이밍 규칙(ORM·마이그레이션 도구가 무엇인지)
7. 횡단 관심사를 확인했다 — DTO·class-validator 검증 규칙, 인증/권한(Guard·전략),
   트랜잭션 경계, 외부 연동(외부 API·큐·캐시), 환경변수/설정
8. **기존 테스트 패턴을 확인했다** — `*.spec.ts`(단위) / `*.e2e-spec.ts`(e2e)의 위치,
   테스트 러너(Jest/Vitest), 모킹 방식, 테스트 DB/픽스처 구성. (TDD 명세 작성에 필요)

## 3단계: 의도 확인 (선택지는 장단점과 함께)

문서에서 다음에 해당하는 항목이 있으면 사용자에게 확인을 요청한다:
- 기존 시스템과 충돌하는 동작 설명
- 여러 가지로 해석 가능한 표현
- 기술적으로 구현이 어렵거나 대안이 필요한 부분
- DB 스키마 변경이 필요한 경우 (특히 파괴적 변경: 컬럼 삭제·타입 변경·NOT NULL 추가)
- **성능·동시성·멱등성 점검 결과 고려가 필요한 지점** (요구 문서에 언급이 없어도, 위
  "백엔드 특이사항 4번" 점검에서 도출된 항목). 구현 방법을 임의로 정하지 말고 선택지로 제시한다.
- 문서에 없지만 구현 시 결정이 필요한 사항 (에러 응답 형식, 엣지 케이스, 권한 등)

**선택지를 제시할 때는 결론만 통보하지 않고 반드시 옵션별 장단점을 함께 보여준다.**
아래 형식을 사용한다:

```
### [결정 필요] {결정 주제}

| 옵션 | 장점 | 단점 | 권장 |
|------|------|------|------|
| A. {옵션 설명} | {장점} | {단점} | ✅/— |
| B. {옵션 설명} | {장점} | {단점} | ✅/— |

권장: {옵션}, 이유 — {한 줄}
```

모든 확인이 완료될 때까지 명세서를 생성하지 않는다.

## 4단계: 명세서 생성

아래 형식의 명세서를 `Docs/spec/` 디렉토리에 생성한다.

### 파일명 규칙

`[spec]{날짜}_{주제}.md`
- 날짜: YYYYMMDD (오늘 날짜)
- 주제: kebab-case

예: `[spec]20260602_refund-system.md`

### 명세서 형식

```markdown
# {기능명} 개발 명세서

> 요구 문서: {요구 문서 파일 경로}
> 작성일: {날짜}
> 마이그레이션 SQL: {있으면 .sql 파일 경로, 없으면 "없음"}

## 1. 개요

{기능의 목적과 핵심 동작을 2~3줄로 요약}

## 2. 요구사항

### 2.1 기능 요구사항

- [FR-1] 요구사항 설명
  - 상세 동작: ...
  - 조건: ...
- [FR-2] ...

### 2.2 API 계약 (해당 시)

각 엔드포인트마다:

- **{METHOD} {경로}** — {용도}
  - 인증/권한: {필요한 Guard·역할}
  - 요청: {DTO명 / 본문·쿼리·경로 파라미터 스키마}
  - 응답(성공): {상태코드} {응답 스키마}
  - 응답(실패): {상태코드 + 에러 코드/메시지} (예: 400 검증 실패, 409 충돌, 404)
  - 멱등성/트랜잭션: {필요 시 정책}

### 2.3 데이터 요구사항 (해당 시)

- 신규/수정 엔티티(테이블): 테이블명 - 추가/변경 컬럼(타입·NULL여부·기본값)
- 인덱스/제약: {유니크·FK·체크 제약}
- 신규 enum: ...
- **마이그레이션 필요 여부: 필요 / 불필요** (필요하면 "5. 마이그레이션" 참조)

## 3. 영향 범위

### 3.1 수정 대상 파일

| 파일 경로 | 수정 내용 | 사유 |
|-----------|----------|------|
| src/... | ... | ... |

### 3.2 신규 생성 파일

| 파일 경로 | 역할 |
|-----------|------|
| src/... | ... |

### 3.3 관련 모듈/시스템

- {모듈명}: {어떤 영향을 받는지}
- 외부 연동/큐/캐시: {영향 여부}

## 4. 테스트 명세 (TDD)

> backend 구현은 TDD 모드다. 아래 케이스는 coder가 작성할 RED 테스트의 입력이 된다.
> 각 FR은 최소 1개의 테스트 케이스와 매핑한다. e2e는 PHASE 3 통합 점검용으로 분리한다.

### 4.1 단위 테스트 (`*.spec.ts` — task 게이트)

| 테스트 ID | 대상(클래스/메서드) | 시나리오 | 입력 | 기대 결과 | 매핑 FR |
|-----------|--------------------|----------|------|-----------|---------|
| UT-1 | {Service.method} | 정상 | ... | ... | FR-1 |
| UT-2 | {Service.method} | 실패/예외 | ... | {던지는 예외/에러코드} | FR-1 |
| UT-3 | {Service.method} | 엣지 | ... | ... | FR-2 |

### 4.2 e2e 테스트 (`*.e2e-spec.ts` — PHASE 3로 분리)

| 테스트 ID | 엔드포인트 | 시나리오 | 기대 상태코드/응답 | 매핑 FR |
|-----------|-----------|----------|---------------------|---------|
| E2E-1 | {METHOD 경로} | ... | ... | FR-1 |

## 5. 마이그레이션 (DB 변경 시에만)

> spec-writer는 SQL 파일을 **작성만** 하고 **실행/적용하지 않는다.** 적용은 구현·배포
> 단계에서 사람이 프로젝트 마이그레이션 도구로 수행한다.

- SQL 파일 경로: `Docs/spec/migrations/[migration]{날짜}_{주제}.sql`
- UP / DOWN(롤백) 모두 포함
- 파괴적 변경(컬럼 삭제·타입 변경)은 "기획 확인 사항"에서 승인받은 경우에만 포함
```

## 4.5단계: 마이그레이션 SQL 파일 작성 (DB 변경이 있는 경우에만)

2.3에서 마이그레이션이 "필요"로 판정된 경우에만 수행한다. 배경 로직만 변경하거나
스키마 변경이 없으면 이 단계를 건너뛴다.

1. `Docs/spec/migrations/` 디렉토리가 없으면 생성한다.
2. `Docs/spec/migrations/[migration]{날짜}_{주제}.sql` 파일을 Write로 작성한다.
3. 아래 형식으로 UP/DOWN을 모두 작성한다:

```sql
-- migration: {주제}
-- spec: Docs/spec/[spec]{날짜}_{주제}.md
-- 작성일: {날짜}
-- 주의: 이 파일은 명세 산출물이다. 적용은 구현/배포 단계에서 프로젝트
--       마이그레이션 도구로 수행한다. spec-writer는 실행하지 않는다.

-- ==== UP ====
{스키마 변경 SQL}

-- ==== DOWN (rollback) ====
{되돌리는 SQL}
```

4. 명세서 상단 메타데이터와 "5. 마이그레이션"에 이 파일 경로를 기재한다.

**금지:** 마이그레이션 명령 실행, DB 접속, `npx typeorm migration:run`·`prisma
migrate`·`psql` 등 적용성 명령 호출. SQL은 파일로만 남긴다.

# 출력 규칙

- 명세서는 `Docs/spec/` 디렉토리에 마크다운 파일로 생성한다.
- 마이그레이션 SQL은 `Docs/spec/migrations/`에 별도 `.sql` 파일로 작성하고, 절대 실행하지 않는다.
- 기술 용어를 유지하며 한국어 스타일을 유지한다.
- 요구 문서의 의도를 변경하지 않는다. 기술적으로 대안이 필요한 경우 "기술 참고사항"에
  제시하되, 선택지는 장단점 표와 함께 제시하고 최종 결정은 사용자에게 맡긴다.
- 문서에 없는 내용을 임의로 추가하지 않는다. 구현에 필요하지만 문서에 없는 사항은
  "기획 확인 사항"에 기재한다.
- 수정 대상 파일은 실제 코드를 Read로 확인한 결과만 기재한다. 추측하지 않는다.
- 명세서 생성 후 "5단계: 구현 실행 안내"를 수행한다.

## 명세서 품질 기준

- 각 FR은 최소 하나의 구현 위치(파일 경로 또는 신규 파일)와 매핑한다.
- **각 FR은 최소 하나의 테스트 케이스(UT 또는 E2E)와 매핑한다.** 테스트로 표현
  불가능한 FR은 쪼개거나 "기획 확인 사항"으로 올린다.
- 데이터 요구사항은 실제 테이블/엔티티명과 컬럼 타입을 명시한다.
- 문서의 모호한 표현("적당히", "빠르게" 등)은 그대로 옮기지 않고 "기획 확인 사항"으로 이동시킨다.
- "6.4 구현 힌트" 섹션은 반드시 작성한다. 2단계에서 탐색한 진입점·데이터 흐름·참조
  구현을 구조화하여 하류 구현 스킬이 코드 탐색을 반복하지 않도록 한다.
- **성능·동시성·멱등성을 점검했음이 명세에 드러나야 한다.** 고려가 필요한 지점이 있으면
  사용자 동의로 확정한 결정을 해당 섹션(API 계약의 멱등성/트랜잭션, 데이터의 인덱스,
  기술 참고사항)에 반영하고, 점검 결과 불필요하다고 판단했으면 "기획 확인 사항"에
  "성능/동시성/멱등성: 해당 없음(사유)"로 한 줄 남긴다.

## 5단계: 구현 실행 안내

backend 스택의 구현 경로는 `implement-agent`(common 골격 + backend 어댑터, TDD 모드)
하나다. 명세서 생성 후 규모를 한 줄로 요약하고 실행 명령을 안내한다.

### 출력 형식

```
## 명세서 생성 완료

- 명세서: {명세서 파일 경로}
- 마이그레이션 SQL: {경로 또는 "없음"}

### 규모 요약

| 항목 | 내용 |
|------|------|
| 수정/생성 파일 | {n}개 |
| 영향 모듈 | {모듈명 나열} ({n}개) |
| DB 변경 | {요약 또는 "없음"} |
| 테스트 케이스 | 단위 {n}개 / e2e {n}개 |

구현을 진행하려면 아래 명령어를 실행해주세요:

/implement-agent @{명세서 파일 경로}
```

- `{명세서 파일 경로}`는 실제 생성된 파일의 상대 경로로 치환한다.
- 사용자가 실행을 요청하기 전까지 `implement-agent`를 자동 호출하지 않는다.
